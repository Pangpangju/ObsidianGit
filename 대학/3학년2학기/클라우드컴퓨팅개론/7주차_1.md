# 고려사항
### 이벤트 시스템 통신
- 엄청난 자원이 필요한 즉각적 실시간 통신이 손실없이 작동하려면 서비스간 통신이 이벤트 시스템을 통해 작동하여야 한다.
### 대기시간/실시간 자원 고려
- 각 서비스의 동작 형태에 따라 자원 배분과 자원 활용 시간을 고려하여야 한다. 메시지, 실시간 통신, 방송 등은 딜레이 없이 진행되어야 하는 기능이기 떄문에 자원의 활용에 각별한 주의가 필요하다.
- 서버리스 기능은 stateless 하기 때문에 외부 DB를 활용하여 유저 session, 서버 정보, 서버 내 커뮤니티 정보 등을 관리하여야 한다.
### 확장성
- 서버리스 아키텍쳐는 필요에 의해 function 이 작동하기 때문에 디스코드의 방대한 function 의 load에 적합하다. 
- 하지만 용량이 큰 메시지 (동영상, 사진)등은 기본적으로 제한을 두고 horizantal scaling을 사용해야 한다. 
### 요금폭탄
- 서버리스 아키텍쳐는 요청의 수와 실행시간에 비례해 작동하기 때문에 유저들의 평소 행동을 잘 관찰하여야 한다 (Logging)
- Cold start time 을 최적화하는 것도 하나의 방법이다. AWS Lambda의 Provisioned Concurrency Feature 기능을 활용한다.
# 독립적 동작 이유
- 서버리스 컴퓨팅의 각 function 은 stateless 하고 event-driven 이기 때문이다.
- 독립적으로 작동함으로써 서버리스 컴퓨팅은 확장성, 유동성을 기대할 수 있고, Fault Isolation을 통해 오류가 어디서 발생하였는지 쉽게 확인할 수 있다.
# 장단점
### 장점
- Idle 자원 X
	- 요청을 받을때만 작동을 하기 떄문에 기존의 컴퓨팅 설계와 달리 idle 자원이 없다.
- 비용절감
	- 사용한 자원에 대해서만 가격이 측정된다
- 독립적 동작으로 인한 높은 확장성과 오류 감지 능력

### 단점
- Cold Start
	- Request가 항상 있는 것이 아니기 때문에 피할 수 없는 문제중 하나이다.
- Stateless 특성
	- 독립적이고 stateless한 특성 덕분에 각각 서비스와 task 들 사이에 지속적인 데이터 연결이 필요한 경우에 문제가 생길 수 있다.
- 보안 문제
	- 클라우드 환경에서 여러 tenant들 사이에서 공유되기 때문에 당연히 보안 문제가 생길 수 밖에 없다.

# 보안문제
- API 보안 강화
	- Authentication Layer 추가
- 데이터 암호화
	- DB나 클라우드 저장소에 있는 정보들을 암호화
- 각 function의 캡슐화 및 isolation
	- 클라우드 function 설계에 있어 각각 function들 사이에 연결을 미약하게 만들어야함
- Event 시스템 보안 강화
	- 이벤트 통신에 보안 layer를 추가한다.
	- Secure messaging queues, API 게이트웨이 보안 등...
- 모니터링
	- 각각 기능에 대해 유저의 요청과 자원 활용을 지속적으로 모니터링 해 보안상의 허점은 없는지 아키텍쳐에 해를 가할만한 요소는 없는지 검사하여야 함
